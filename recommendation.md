# Recommendations: top 50 things done badly or wrong

This is a deduplicated, severity-ranked shortlist drawn from 13 review lenses over the Wheel of Shame frontend and its Rust backend. Cross-lens duplicates were merged to the sharpest phrasing (for example the empty-`sessionId` history bug surfaced in five lenses and the double `spin-complete` emit in four), the junk placeholder finding was dropped, and items already landed by the merged `feature-batch-craft` work (WheelAudio decomposition, type-contract extraction, opentype chunk removal) are excluded. What remains is dominated by features that look wired but are dead (history/undo, winner reveal, weight preview, presence), realtime/state races that corrupt the shared roster, GPU resource leaks in the Three.js renderer, a near-total absence of accessibility on the central canvas and modals, and a backend with no auth, no CORS allowlist, and broken time-travel semantics. Effort tags: S (under ~1h), M (a few hours), L (a day or more).

**Severity tally (top 50):** Critical 4 | High 22 | Medium 19 | Low 5

## Critical

1. **History/undo feature is permanently dead: `useHistory('')` closes over an empty session id** [critical] - `useHistory.ts:23,28,42,70,85` + `useSession.ts:18-22,307,332`. `useHistory` captures `sessionId` by value, constructed once as `useHistory('')`; `setHistorySession` writes a separate `_sessionId` field nothing reads, so every method early-returns on `if (!sessionId)` forever. Impact: `loadActions`, `undo` (Cmd-Z), `restoreTo`, and history-scrub `preview` all silently no-op, and `replayEvents` is never fed real actions, so the entire server-backed history/undo/restore feature is non-functional while appearing fully wired. Fix: change the signature to `useHistory(sessionId: Ref<string>)` (or a getter), read `.value` in every method, and have `setHistorySession` update that ref; delete the `_sessionId` cast hack. (M)

2. **`clearGroup` never disposes per-segment label textures, leaking a `CanvasTexture` per name on every rebuild** [critical] - `ThreeWheelRenderer.ts:clearGroup` + `build` + `makeSimpleLabel`. Labels are added as children of the segment mesh, but `clearGroup` only iterates direct children of `wheelGroup`, so each label's `PlaneGeometry` + `MeshBasicMaterial` + `CanvasTexture` is orphaned. Impact: `build()` runs on every roster-size and committed-weight change, leaking one GPU texture per participant per rebuild, unbounded VRAM growth that degrades FPS and can crash the WebGL context on large rosters over a session. Fix: traverse nested children in `clearGroup` (or dispose labels explicitly before clearing) and call `material.map?.dispose()`; better, cache label textures by `name|color|dpr` and mark them shared so they survive rebuilds. (M)

3. **`clearGroup` disposes cached/shared materials and geometries that the next `build()` reuses** [critical] - `ThreeWheelRenderer.ts:build` + `clearGroup` + `getCachedMat`/`getSegmentGeometry`/`getPegMat`. Segment meshes use cached material and geometry but set neither `userData.sharedMaterial` nor `sharedGeometry`; pegs set only `sharedGeometry`. So `clearGroup` calls `.dispose()` on resources still referenced from the caches. Impact: after the first rebuild the renderer hands out disposed `MeshStandardMaterial`/`ExtrudeGeometry`, producing black renders or "attempt to use a deleted object" WebGL warnings and silently defeating the caching layer after any roster edit. Fix: set `sharedGeometry`/`sharedMaterial` on the segment and peg meshes so `clearGroup` skips them, disposing the caches only in `dispose()`. (S)

4. **Global Tab handler hijacks Tab into wheel-segment cycling, breaking standard focus traversal** [critical] - `useHotkeys.ts:onGlobalKeydown` (Tab branch). The window-level handler `preventDefault`s bare Tab/Shift+Tab whenever focus is outside an input and the roster is non-empty, repurposing Tab to cycle a CSS-only `focusedId` spotlight that is not real DOM focus. Impact: a WCAG 2.1.1/2.1.2 failure; once focus rests on the body (after a modal closes or on load), a keyboard or screen-reader user can no longer Tab to the dock, name input, remove buttons, or any panel, and the "focused" segment cannot even be activated. Fix: stop calling `preventDefault` on bare Tab; make the segment spotlight a genuine roving-tabindex/listbox widget driven by arrow keys, leaving Tab to the browser. (L)

## High

5. **Renderer never fires `onWinner`, so the winner modal and screen-reader reveal are dead** [high] - `ThreeWheelRenderer.ts:_onWinner` (stored at :172, only `void`-referenced at :196). The callback is stored but never invoked; `winnerData` (the modal gate) has no other writer. Impact: after every spin the `SpinResultModal` never appears and the assertive `winnerAnnounce` aria-live region never speaks the eliminated name, so the app's central payoff silently does nothing for sighted and screen-reader users alike. Fix: call `this._onWinner({ id, name, remaining })` in `playSpin`'s settle path, resolving name/remaining from the segment meshes; add a regression test asserting one invocation per completed spin. (S)

6. **`replayEvents` wipes the real server roster and rebuilds it with synthetic `replay-*` ids on load** [high] - `roster.ts:replayEvents` (42-62), invoked from `useSession.ts:310,335`. After `setParticipants(data.participants)`, load folds the action log into `participants.value = []` then re-adds rows with fabricated `'replay-'+a.id` ids that never match server ids. Impact: any session with logged actions loads into a corrupted state where later spins/removes/weight edits (carrying real ids) cannot find their target and WS echoes duplicate rows. Fix: do not call `replayEvents` after `setParticipants` on load (the server projection is already folded), or reconcile against existing rows by real id instead of resetting to `[]`. (M)

7. **A failed spin leaves `suppressWsSpin` stuck true, swallowing the next remote spin** [high] - `useSession.ts:doSpin` (sets flag at :369) + `spin_result` handler (resets only at :243). `doSpin` sets the module-level `suppressWsSpin = true` before `await api.spin()`; the catch returns without resetting it, and only an incoming `spin_result` clears it. Impact: after any failed local spin, the next genuine `spin_result` from any client is treated as the user's own echo and dropped, so a remote elimination never animates and the two sessions diverge until another spin arrives. Fix: reset `suppressWsSpin = false` in `doSpin`'s catch; better, scope suppression to a specific spin id rather than a sticky boolean. (S)

8. **`playSpin` lands using uniform slices while the wheel is built with weighted arcs, so it stops on the wrong name** [high] - `ThreeWheelRenderer.ts:playSpin` (482-488) vs `build` (315-332). `build` lays out arcs by `p.angle` (non-uniform under weights), but `playSpin` computes the landing target and the per-frame tick from `(2*PI)/activeCount` uniform slices, with a different pointer origin. Impact: with non-equal weights (the editor's whole purpose, plus AI/equalize/templates) the wheel visually stops on a segment that is not the server's winner and the ticking highlight lights the wrong row, so the animation contradicts the recorded outcome. Fix: compute the landing angle from the cumulative weighted prefix-sum used in `build()` and feed the same per-segment boundaries into the tick mapping. (M)

9. **`WheelCanvas` drops `previewOverrides` on the floor: weight preview never reaches the 3D wheel** [high] - `WheelCanvas.vue:14` (prop) + `ThreeWheelRenderer.ts:49,434` (`previewOverrides`/`updateInPlace`). App binds `:preview-overrides`, but the component never forwards it; `this.previewOverrides` is never assigned and the only reader, `updateInPlace`, is never called. Impact: dragging the Inspector weight slider or scrubbing history produces zero change on the wheel, so a core editor affordance gives silent no-op feedback. Fix: add a `setPreviewOverrides()` to the `WheelRenderer` interface, forward the prop via a `watch`, and route preview to the in-place path; or delete the whole preview-on-wheel chain if out of scope. (M)

10. **Double `spin-complete` emit: renderer fires both `onComplete()` and `onSpinComplete()` for one spin** [high] - `ThreeWheelRenderer.ts:577-578` + `WheelCanvas.vue:58,68`. `settle()` synchronously calls both the per-spin `onComplete` callback (which emits `spin-complete`) and the constructor `onSpinComplete` (which also emits `spin-complete`), so `useSpin.onSpinComplete` runs twice. Impact: a latent double-dispatch in the most important callback; today partly absorbed by `pendingSpinResult` nulling, but it re-runs recap-queue logic and will double-execute any non-idempotent side effect (double history, analytics, removal on the remote/in-flight path). Fix: emit from exactly one channel and make `useSpin.onSpinComplete` idempotent; add a test asserting one emit per spin. (S)

11. **`dispose()` leaks the EffectComposer/bloom render targets and the PMREM environment texture** [high] - `ThreeWheelRenderer.ts:dispose` (653-668) + `initScene` (223-241). `dispose()` never disposes `this.composer` (two full-size render targets) and `UnrealBloomPass` (~5 mip targets), nor the PMREM-generated `scene.environment` cubemap. Impact: each `WheelCanvas` mount/unmount (route change, HMR, conditional `v-if`) leaks large GPU allocations, steadily exhausting VRAM on repeated remounts. Fix: in `dispose()` call `this.composer?.dispose()`, `this.bloomPass?.dispose()`, and `(scene.environment as Texture)?.dispose()`, then null the references. (S)

12. **Window resize listener is added but never removed in `dispose()`** [high] - `ThreeWheelRenderer.ts:constructor` (188) + `dispose` (653-668). The constructor does `window.addEventListener('resize', this.onResize.bind(this))` without storing the bound reference, and `dispose()` never removes it. Impact: every disposed-then-recreated renderer leaves a dangling resize handler closed over a dead scene/composer/camera; on the next resize all stale handlers fire against disposed objects, and the live listener also retains the old renderer graph from GC, compounding the leaks above. Fix: store the bound handler in a field and `removeEventListener` it in `dispose()`. (S)

13. **3D wheel canvas is invisible to assistive tech and has no keyboard-operable spin control** [high] - `WheelCanvas.vue:93` (`.wheel-bg` host) + appended `<canvas>`. The wheel is a bare div hosting a Three.js canvas with no role, label, text alternative, or focusable element, and the spin affordance is a pointer-only `onSpinClick` with no keyboard equivalent on the canvas. Impact: WCAG 1.1.1/4.1.2 failure; a screen-reader user gets no announcement the wheel exists and no way to operate it, depending entirely on the separate roster list and Space key, which are undiscoverable from the canvas region. Fix: wrap the host in a labelled `role="img"` region summarizing state, add a visually-hidden focusable spin button inside it, and mark the canvas `aria-hidden`. (M)

14. **SpinResult and RecapReel modals have no dialog semantics, focus trap, or focus restoration** [high] - `SpinResult.vue:13-25` + `RecapReel.vue:209-256`. Both are teleported full-screen overlays with no `role="dialog"`/`aria-modal`, no focus move-in, no Tab trap, and no focus restore on close; SpinResult also lacks a label. Impact: WCAG 2.4.3/4.1.2; the winner reveal (the app's emotional climax) is announced as nothing, Tab walks the obscured page behind the scrim, and after Continue/Done focus is lost to body. Fix: add `role="dialog" aria-modal="true"` + label, focus the primary button on mount, trap Tab, and restore focus to the trigger on close (reuse the CommandPalette/ShortcutsSheet pattern). (M)

15. **CommandPalette and ShortcutsSheet declare `aria-modal` but never trap or restore focus** [high] - `CommandPalette.vue:291,266-285` + `ShortcutsSheet.vue:30-37`. CommandPalette binds keydown only to its input and implements no Tab trap or focus restore; ShortcutsSheet never moves focus in on open. Impact: WCAG 2.4.3; `aria-modal="true"` tells AT the background is inert, yet Tab still escapes into it, producing a mismatch where the user interacts with controls the screen reader claims are hidden, and closing drops focus to body. Fix: trap Tab/Shift+Tab within each dialog, focus the first interactive element on open, and restore focus to the invoking control on close. (M)

16. **Every roster change tears down and rebuilds the entire Three.js scene; `updateInPlace` is dead** [high] - `WheelCanvas.vue:50` (deep watch -> `build()`) + dead `ThreeWheelRenderer.ts:434`. The only reaction to `preparedSegments` is a full `clearGroup` + reallocate of N meshes/lines/pegs/donut arcs plus a fresh canvas+`CanvasTexture` label per name; the documented fast path `updateInPlace` is never called. Impact: on a 30-name roster a single slider nudge or history-scrub re-rasterizes 30 textures and rebuilds ~150 meshes, and during a live weight-preview drag this fires per animation frame, causing visible jank and GC churn (and triggering the leaks above). Fix: diff `preparedSegments` and call `updateInPlace()` for weight/color-only changes, reserving `build()` for structural (count/order/id-set) changes. (M)

17. **`refreshPreparedSegments` has no request-sequence guard: stale worker results overwrite fresh ones** [high] - `roster.ts:257-283` + watch at :286. It `await`s `computeAnglesAsync` then unconditionally assigns `preparedSegments.value`; rapid edits launch overlapping async calls resolved purely by message id, so an older call resolving last clobbers the newer roster. Impact: the wheel and NameList odds can render angles for a roster state that no longer exists (wrong slices, an already-removed name, a length mismatch that silently disables NameList's odds fast path), an intermittent load-dependent bug that survives manual testing. Fix: add `let reqSeq = 0`, capture `const seq = ++reqSeq` before awaiting, and bail `if (seq !== reqSeq) return` before assigning. (S)

18. **WheelEngine pick/bias/simulate cluster is entirely dead, yet synced on every mutation** [high] - `wheelEngine.ts:pick/simulate/suggestLeastRecentBias/equalize/reorder/setBias` + `roster.ts:syncEngine`. The server decides winners and the worker computes angles; none of the engine's pick/bias/simulate methods are called, and `getPreparedSegments` is only a sync fallback, yet `syncEngine()` runs on every roster change to keep a dead second copy of the roster in lockstep. Impact: ~60% of a class advertised as "the domain core" is unreachable and duplicates server-side picking, so a contributor can wire `engine.pick()` and silently reintroduce a client winner that disagrees with the authoritative result. Fix: delete the dead methods and `biasMap`, keep only `setSegments/setPreventRepeat/setLastPicked/getPreparedSegments`, rename the class to something honest, and stop calling `syncEngine()`. (M)

19. **Roster Pinia store instantiates component-lifecycle composables as hidden internal state** [high] - `roster.ts:20-21` (`usePreventRepeat`/`usePreview`) + `usePreview.ts:15,25`. The store calls these at setup, folding their refs into its surface; `usePreview` registers `onBeforeUnmount` and owns a rAF throttle, but a Pinia store is a long-lived singleton outside any component, so the cleanup binds to an arbitrary or no component. Impact: a domain store depends backwards on Vue component lifecycle, the preview rAF can leak or be cancelled by an unrelated unmount, and prevent-repeat state is duplicated (store + composable) - the dual-source-of-truth the refactor meant to kill. Fix: inline `previewOverrides`/`preventRepeat`/`lastPickedId` as plain refs with a module-level throttle handle and delete the composable instantiation from the store. (M)

20. **`useSession` is a module-level god-singleton mixing transport, reconnect, toasts, recents, and cross-store reach-in** [high] - `useSession.ts:12-105,93-207,209-291`. Everything but the `useSession()` factory lives at module scope: the WebSocket, the full backoff reconnect state machine, recents persistence, and a 9-case `ws.onMessage` switch that calls `useRosterStore()` and rebuilds arrays inline in nearly every case. Impact: single-responsibility is gone (a reconnect or toast tweak risks the others), the process-global singleton can never be reset or tested in isolation, and roster mutation logic is duplicated between this WS path and `roster.ts` (which already disagree). Fix: split into `useWebSocketSync`, a reconnect-policy module, and `useRecentSessions`; have the WS handler call named roster actions instead of rebuilding arrays. (L)

21. **`getSnapshot(before=actionId)` orders by random UUID (memory) or ignores `before` entirely (sqlite)** [high] - `backend/store/memory.rs:237-246` + `sqlite.rs:558-574`. The contract promises "snapshot at or immediately before the action," but `action_id` is a random v4 UUID with no chronological order, so the memory store's lexical `<=` returns an arbitrary snapshot and the sqlite store ignores the arg and always returns the latest. Impact: `restoreTo`, `previewFromHistory`, and `undo` request a specific historical point but receive an arbitrary or current snapshot, so time-travel restore is non-functional and silently restores the wrong state. Fix: give snapshots/actions a monotonic ordinal (rowid or timestamp), resolve `before` against that ordinal in both stores, and test that two spins then `getSnapshot(before=first)` returns the one-removed state. (M)

22. **Spin snapshot stores the post-spin (already-removed) state under the spin action id** [high] - `backend/store/memory.rs:148-165` + `sqlite.rs:377-393`. The picked participant is marked `removed` before the snapshot is built, and the snapshot is keyed to the spin action's own id with the post-removal list (the pre-state `_before_snapshot` is built and discarded). Impact: undoing a spin (the primary thing users want to undo) restores the state in which the victim is already eliminated, so the entire snapshot-per-spin scheme cannot reverse a spin. Fix: snapshot the pre-mutation list and associate it with the action under a single consistent "state before action X" convention honored by spin, restore, and `get_snapshot`. (M)

23. **No CI pipeline: tests, type-check, and build never run automatically** [high] - repo root (no `.github/workflows`, no CI yaml). `package.json` exposes `test`/`build` but nothing wires them to push/PR; `vue-tsc` runs only as a side effect of `build`, and the 39 vitest tests run only when a human types `npm test`. Impact: regressions ship silently, a broken type or failing test can merge to `main` unguarded, undermining the entire existing suite exactly during a mid-flight god-class decomposition. Fix: add `.github/workflows/ci.yml` running `npm ci`, `npm run build`, and `npm test` on PR and push to main; add standalone `typecheck` and `lint` scripts so each stage reports independently. (S)

24. **Test environment is `node` with no DOM, so the entire component layer is structurally untestable** [high] - `vite.config.ts` (`test.environment: 'node'`) + `package.json` devDeps. No jsdom/happy-dom, no `@vue/test-utils`, no `setupFiles`; the 12 `.vue` components and every composable touching `window`/`localStorage`/lifecycle/`WebSocket` cannot be mounted. Impact: coverage is permanently capped at the pure-util slice no matter how many tests are written, and any author who tries to mount a component hits a wall (no `document`). Fix: add `happy-dom` and `@vue/test-utils`, set `test.environment: 'happy-dom'`, and add a `setupFiles` registering Pinia. (M)

25. **XSS via unescaped title and color interpolated into the raw SVG export** [high] - `utils/export.ts:19-23`. `exportWheelSVG` string-concatenates the session `title` into a `<text>` body and `color_override` into a `fill` attribute with no escaping; both are attacker-controllable (title length-checked only, color a free-form server-persisted string). Impact: stored XSS in the exported artifact - a malicious participant can poison the title or a segment color so anyone who opens the downloaded `.svg` runs attacker JS (cookie/localStorage theft); this is the only place bypassing Vue's auto-escaping. Fix: HTML-escape `title`, validate `color_override` against a strict color pattern with a safe fallback, or build the SVG via DOM APIs so the browser escapes. (S)

26. **No authorization on any session endpoint: the session id is a bearer capability with full write access** [high] - `backend/routes.rs` (all routes) + `handlers.rs` spin/reset/delete + `models.rs:88` (`actor: None`). Every mutating route is wired with no auth layer, and the id leaks via `location.hash`, copyLink, history, and referrers. Impact: anyone who learns or guesses an id can spin, delete participants, reset, or restore arbitrary state for a live shared room, with no audit trail of who did it - a real integrity/abuse problem for an app that publicly assigns blame. Fix: on create, return an unguessable edit token separate from the readable id and require it (header/cookie) for mutating routes via an axum extractor; populate `Action.actor`. (L)

27. **`restore_from_snapshot` overwrites server state with unvalidated client-supplied participants** [high] - `backend/handlers.rs:337-364` + `store/memory.rs:248-259`. The `/restore` handler takes a client `Vec<Participant>` and does `*slot = participants.to_vec()` with zero validation, unlike add (trim + length) and update (weight clamp). Impact: a client bypasses every server-side input constraint - arbitrary ids, 10MB names, non-finite weights that break the spin math, or HTML/SVG payloads that chain into the export XSS - and with no auth any link holder can corrupt the room. Fix: run restore participants through the same validation as add/update (trim, length, clamp weight to [0.1,10], reject non-finite, cap array length, verify ids belong to the session). (M)

28. **Optimistic `addName` failure is swallowed in the store; the error ref and the user never learn** [high] - `roster.ts:addName` (166-200) + `useSession.ts:351`. The store's catch marks the temp row errored, schedules a 600ms removal, and returns normally without rethrowing or toasting, so `useSession`'s `.catch(e => error.value = e.message)` is dead code. Impact: on a rejected add (duplicate, validation, 500, offline) the name silently flickers and vanishes with zero feedback; the red error banner is wired to a path that can never fire for adds. Fix: in the store catch, push an error toast or rethrow so `error.value` populates; apply one consistent contract across `addNames`/`removeName` too. (S)

## Medium

29. **Peer presence count is `Math.random()` re-rolled on every render** [medium] - `App.vue:705`. The peers chip interpolates `Math.floor(Math.random()*3)+2` inline in the template, recomputed and re-randomized on every reactive re-render (hover, tick, spin, toast), while the real `ui.presence` map is never read for the count. Impact: users see a peer count that jumps randomly on unrelated interactions and never reflects reality, misrepresenting the core "who's in the room" signal of a collaborative app and undermining the adjacent "Live" status. Fix: drive the count from real WS join/leave data feeding `ui.presence`, or show a static value/hide the chip; never call `Math.random()` in a template binding. (S)

30. **Canvas click starts a spin on any click, including the mouse-up ending an OrbitControls drag** [medium] - `ThreeWheelRenderer.ts:onCanvasClick` (624-632); `_isPointerDown` tracked but never consulted. `onCanvasClick` unconditionally calls `onSpinClick`, and an OrbitControls camera drag ends with a click event, with no drag-vs-click threshold and no guard while a modal is open. Impact: rotating the camera by dragging the wheel triggers an unintended spin on release, so a user inspecting the wheel from another angle accidentally eliminates someone. Fix: record pointerdown position/time and only fire if pointerup is within a small movement and time threshold, and ignore clicks while spinning or while a winner/recap overlay is open. (S)

31. **Module-scope composables run at import time, outside any component or Pinia scope** [medium] - `useHistory.ts:7` + `useSession.ts:10,196-291` + `useVoice.ts:3`. `useToasts()` and module-level refs plus `ws.onMessage`/`onOpen` handlers execute at import, before any app/Pinia is mounted, and the WS handlers call `useRosterStore()` at message time without the active-Pinia guarantee a `setup()` gives. Impact: any second mount or test shares the same session/ws/history singletons with no teardown, `onBeforeUnmount` cleanups never fire, and store getters inside WS handlers are fragile across tests (no `setActivePinia`). Fix: move singleton creation into a guarded factory created on first `useSession()` call inside a component, and register WS listeners with matching teardown. (L)

32. **`preparedSegments` watcher keys on `JSON.stringify` and ignores prevent-repeat/lastPicked changes** [medium] - `roster.ts:286` + 272-273,303-306. The only recompute trigger serializes the weight array on every reactive tick and keys purely on length+weights, while `isLast`/`isPrevented` derive from `preventRepeat`/`lastPickedId` whose setters call only `syncEngine()`, never `refreshPreparedSegments()`. Impact: those flags go stale until an unrelated edit retriggers the watcher, and the stringify allocates a string per tick on the hottest path; the next consumer to trust the flags inherits a silent bug. Fix: replace the stringify key with a structured watch source, add the prevent/lastPicked sources, and call `refreshPreparedSegments()` from the toggle/setLastPicked actions. (M)

33. **WS handlers replace the whole roster via `setParticipants` on every event, racing optimistic temp rows** [medium] - `useSession.ts:210-289` + `roster.ts:93-96,166-200`. Every WS branch rebuilds the array and full-replaces via `setParticipants` (deep-clone + `syncEngine`); a `participant_added` event arriving between an optimistic insert and its REST resolve replaces the array identity so the temp-row reconciliation indexes the wrong generation. Impact: concurrent adds can leave duplicated names or a stuck pending skeleton, and every event discards referential stability, forcing a full WheelCanvas deep-watch rebuild. Fix: make roster mutations granular (`addOne`/`removeOne`/`upsertById`) keyed on id, and dedupe the server echo by id before splicing. (L)

34. **App.vue `watch(removedParticipants)` fights `useSpin`'s recap queue and can swallow the recap** [medium] - `App.vue:377-382` + `useSpin.ts:36-39,45-51`. Two owners manage recap lifecycle: `useSpin` queues/opens it, while an App watcher force-clears both flags on the empty transition. Impact: on reset/session-switch the watcher stomps recap state, and combined with the duplicated spin-complete the ordering is nondeterministic, producing occasional "final pick but no recap" or a `recapQueued` stuck true that pops on the next dismiss. Fix: give `useSpin` a single `resetRecap()` called from the reset flow and drive recap from one owner keyed off an explicit reset event, not inferred from `removedParticipants.length === 0`. (M)

35. **`create()`/`load()` fire-and-forget history replay: unhandled rejection and post-switch clobber** [medium] - `useSession.ts:308-312,333-337`. Both call `history.loadActions().then(() => roster.replayEvents(...))` un-awaited and without `.catch`. Impact: a throwing `replayEvents` surfaces only as an uncaught promise rejection, and a late-resolving replay from a previous session can overwrite a freshly loaded roster after a quick session switch. Fix: await the replay inside the try so the existing catch handles it, capture the session id at call time, and bail if `session.value?.id` no longer matches (mirroring the existing reconnect-gen guard). (M)

36. **`useHistory` undo/restore/preview swallow every error with `console.warn`, masking real failures** [medium] - `useHistory.ts:41-96`. Each catch only `console.warn`s; `restoreTo` additionally does `api.restoreFromSnapshot(...).catch(() => {})` and then toasts success regardless. Impact: the user clicks Undo/a history entry and nothing happens with no feedback, and a half-applied restore (local changed, server unchanged) is reported as success. Fix: replace the warn-only catches with a user-facing toast, await `restoreFromSnapshot` inside the try, and toast success only on resolution (toast failure and re-sync on rejection). (M)

37. **AnalyticsPanel `Avg` is a removed/total ratio (always 0-1) mislabeled as average spins** [medium] - `AnalyticsPanel.vue:29,60`. `avgSpins = removedCount/total` is the eliminated fraction, displayed as "Avg" alongside Total and Removed, implying an average-spins-per-name metric. Impact: the analytics surface reports a meaningless always-sub-1 number under a misleading label (a 10-person roster with 3 out shows "Avg: 0.30"), which is misinformation. Fix: remove the metric or relabel and rescale it correctly (e.g. "Eliminated: 30%"). (S)

38. **Frontend invents a `Comment` ActionKind the backend enum does not have** [medium] - `types/index.ts:45` + `App.vue:161-173` vs `models.rs:91-102` + `sqlite.rs:521-531`. The TS `ActionKind` union has a `Comment` variant pushed through `recordAction`, but the Rust enum has none, and sqlite's `list_actions` match falls through unknown kinds to `_ => ActionKind::Reset`. Impact: client and server disagree on the action vocabulary, comments masquerade as log actions, and any unknown kind round-tripped through sqlite is silently coerced to a data-corrupting Reset. Fix: move comments out of the `Action` type into a client-only type (or add a real `Comment` variant with persistence), and replace the silent `_ => Reset` fallback with an explicit error/Unknown. (M)

39. **WS spin dedup relies on a single module-level boolean against the server echoing your own spin** [medium] - `useSession.ts:79,240-263,366-380` + `backend/handlers.rs:175-193`. The server broadcasts `SpinResult` to all subscribers including the initiator, and the client distinguishes its own echo from a peer's spin only via the sticky `suppressWsSpin` flag with no correlation id.
